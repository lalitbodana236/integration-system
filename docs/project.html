project deep-dive

The interview narrative for the integration platform.

This page condenses the deep-dive into something you can skim before an interview, then use the simulator to show the same story live.

At a glance

HTTP ingressIngestion accepts /api/po, /api/so, /api/inventory, /api/checklist, /api/location, and /api/media.
Kafka backboneRaw, retry, region, and DLQ topics move work asynchronously between services.
Idempotent deliveryDuplicate protection exists at ingestion, processing, regional, and notification layers.

Service responsibilities

Service What it does Interview angle
ingestion-service Receives HTTP requests, assigns correlation IDs, dedupes on eventId, and publishes to raw-events. Fast 202 ACK, idempotency, request normalization.
processing-service Consumes raw-events, validates and persists state, then routes by region to region-india/us/uk. Batch listener trade-offs, retries, and routing logic.
regional-service Applies region-specific workflow and emits notification-events. Localized workflow isolation and resilience.
notification-service Consumes notification-events and fans out to EMAIL, SMS, and WEBHOOK. Strategy-based fan-out and channel-level idempotency.

API contract highlights

  • POST /api/{type} accepts id, optional customerId, and optional region.
  • Success returns 202 Accepted with status: ACCEPTED and topic: raw-events.
  • Duplicate requests still return 202 with status: DUPLICATE.
  • /api/media uses multipart form-data with a required file and request id.

Retry and DLQ story

  • Processing retry chain: retry-processing-5s → retry-processing-1m → retry-processing-10m.
  • Regional retry chain: retry-region-5s → retry-region-1m → retry-region-10m.
  • Notification retry chain: retry-notification-5s → retry-notification-1m → retry-notification-10m.
  • DLQ payloads preserve payload, stacktrace, retry count, correlation id, and source topic.

Why the design works

ResilienceFailures stay isolated and recoverable through staged retries instead of blocking the whole request path.
ObservabilityCorrelation IDs and event IDs make asynchronous debugging and demo narration much easier.
ScaleEach service can scale independently, which is exactly the interview story this platform is built to tell.
Portfolio valueThe simulator turns an architecture discussion into something visual, interactive, and memorable.

Interview cheat sheet

Why Kafka?Decoupling, buffering, replayability, and independent scaling across service boundaries.
How do duplicates stay safe?Idempotency is layered, so each stage protects its own side effects before writing or dispatching.
Why retries via topics?Backoff is visible, controlled, and easy to reason about without tight synchronous loops.
What does the demo show?A polished end-to-end story from request acceptance to notification fan-out and failure handling.
